# cursor-bridge 流式逻辑说明（含历史去重参考）

---

## 当前策略（只推 result，无去重）

**自 2025 年起**，stream-parser 已**移除去重逻辑**，采用更简单策略：

- **只推送 `type: result` 的整段内容**：解析到 `type === 'result'` 且 `result` 字段有效时，将该段正文原样 push 给客户端；**不转发** `assistant`、`message` 等类型的行。
- **无 result 时兜底**：若整条流从未 push 过任何内容（例如只有 thinking/heartbeat、或仅有 assistant 无 result），在 flush 时推一条用户可见提示（解析异常 / 无可展示内容），避免界面完全空白。
- **重复问题**：不再在桥侧做事件级、跨行 trim、单行内去重；若上游同一条流内既发 assistant 又发 result 导致「两遍」，由 Cursor Agent / 产品侧负责；桥只做 NDJSON → SSE 与类型过滤。

效果：实现简单、无调参；用户端收到的是「整段 result」，无增量流式打字效果，需等待 result 到达后一次性展示。

## 以下为历史去重策略（已废弃，仅作参考）

下文原为「去重逻辑梳理」，保留作排查或对比用；**当前代码已不包含这些逻辑**。

### 结构性梳理：就两种情况——单行内 / 跨行

重复只可能发生在两种范围里：

| 范围 | 重复发生在哪 | 我们怎么做 |
|------|--------------|------------|
| **单行内** | 同一行（同一段 NDJSON 正文）里，同一段话出现两遍 | 只保留第一段再推，把末尾的重复段删掉 |
| **跨行** | 当前这一行和「已经推过的内容」之间有重复 | 再分两种：**整行重复**→整行不推；**只是前缀重叠**→只推新部分，重叠前缀不推 |

处理顺序：先做**跨行**判断（整行重复就 skip，否则 trim 出「新部分」）→ 对**即将要推的那一段**做**单行内**检测（若这一段内部有结尾=开头，只保留第一段）→ 再 push。

---

### 单行内：「结尾 = 开头」是什么意思？（具体例子）

这里的「结尾」「开头」指的是**同一条字符串**的头和尾，不是两行之间的比较。

**例子**：Cursor 某一行的 content 是下面这一整段（一行里就两遍同一句）：

```
我无法读取 Cursor 为本次会话分配的模型名，没有接口能拿到这个信息。我无法读取 Cursor 为本次会话分配的模型名，没有接口能拿到这个信息。
```

- **这条字符串的「开头」**：前 N 个字，例如「我无法读取 Cursor 为本次会话分配的模型名，没有接口能拿到这个信息。」（第一句）
- **这条字符串的「结尾」**：后 N 个字，和上面**完全相同**（第二句）

也就是说：**同一行内容的「结尾一段」和「开头一段」字面相同**，说明这一行里把同一段话打了两次。我们就把**末尾那段重复的删掉**，只推第一段，用户就只看到一句。

**再举个更短的例子**（仅说明概念，实际我们会要求重复段 ≥40 字、总长 ≥80 字才删，避免误伤）：

- 一行内容是：`你好世界你好世界`
- 开头 4 字 = `你好世界`，结尾 4 字 = `你好世界` → 相同
- 我们就只保留并推送 `你好世界`，不推两遍

---

### 跨行：两种处理方式

- **整行重复**：当前行和已发内容完全一样、或已发已经包含当前行、或当前行 = 已发 + 很短尾巴（如一句收尾）。→ **整行不推**（事件级 skip）。
- **前缀重叠**：当前行 = 已发内容的尾部 + 新内容（例如已发过「问1」，这一行是「问1问2」）。→ **只推新部分**「问2」，重叠的「问1」不推。

---

## 字数限制是不是必须的？

不是每个情况都「必须」用字数，但用了是为了**性能**和**误判控制**。

- **单行内**  
  - 「结尾 L 字 = 开头 L 字」里，我们要求 L 在 40～半长、且整段 ≥80 字才检测。  
  - **为啥**：太短（如 L=5）会把正常排比当重复误删；只对「整段级」重复动手。  
  - **能不用字数吗**：可以改成「只要结尾=开头就删」，但误伤会多；40/80 等价于「只认整段重复」。

- **跨行**  
  - 8、20、600、2500、25 等：控制「多短不判重复」「用已发多长的尾巴做比较」「多短的尾巴算整行重复」等，都是为了性能和不误杀。  
  - 若用更语义的规则（如「足够长」「尾巴足够短」）也可以，字数只是当前实现方式。

## 参考：cursor-agent 为啥没有「去重」问题？

**cursor-agent**（OpenClaw 插件，同仓库下的 `cursor-agent`）也消费 Cursor CLI 的 stream-json，但**不做去重**、界面上也不会出现两遍，原因在于**使用方式不同**：

| 维度 | cursor-agent（插件） | cursor-bridge（桥） |
|------|----------------------|----------------------|
| **输出方式** | **先收齐再一次性出**：跑完 CLI，收集整条流的 events，再调 `formatRunResult` 生成几条消息 | **边收边推**：每行 NDJSON 解析后立刻转成 SSE 推给客户端，实时流式 |
| **正文从哪来** | `buildConclusion(events)` 只取 **最后一个** `type: "assistant"` 的 `text`，**不拼接**所有 assistant，也**不用** result 当正文 | 必须把**每一段** assistant/result 推出去，客户端自己拼成一条消息；若上游同一段既发多段 assistant 又发 result，就会重复 |
| **result 用在哪** | result 只用来填 `RunResult.resultText`、usage 等，**不参与**展示用的「结论」文案 | result 和 assistant 一样会当 content 推给前端，所以会和多段 assistant 一起形成「同一段话两遍」 |

所以：**插件是「只取最后一条 assistant 当结论」**，天然没有「多段 assistant + result 拼成两遍」的问题；**桥是「逐块转发」**，必须自己做跨行/单行去重，才能避免重复。

## 「流式」下能不能只推「最后一条完整回复」？

**可以。**「最后一条完整回复」在 Cursor 的 stream-json 里是相对确定的位置：通常是 **`type: result`** 那一行（或流结束前最后一条带完整正文的 assistant）。  
若桥改成：**不转发中间的 assistant 块，只等 result 到了再推 result 的内容**（或流结束时用最后一条 assistant），就等价于「只推最后一条完整回复」，**不需要任何去重**，也不会出现两遍。

**那为什么桥没有这么做？**  
因为当前桥的设计目标是**增量流式**：每来一块 assistant 就推一块，让前端能「边生成边出字」。若改成「只推 result」：

- 用户端体验会变成：请求发出后先等一段时间（可能几十秒），然后**一次性**出现整段回复，中间没有逐字/逐句的更新。
- 技术上没有「不能」，是**产品选择**：要增量流式 → 必须转发 assistant 块 → 就会遇到上游多发重复 → 所以要做去重；若接受「等完再一次性出」，就可以不转发 assistant、只推 result，也就不需要去重。

**小结**：  
- 「最后一条完整回复」有确定来源（result 或最后一条 assistant）。  
- 流式场景下**可以**只推它（不推中间块），但会失去「边生成边显示」；要保留增量显示，就得转发 assistant 并做去重。

## result 是一点点返回还是整段返回？

**整段。** Cursor Agent 的 stream-json 协议里：

- **assistant**：会发**很多行** NDJSON，每行一小块正文（十几字到几十字），所以是「一点点」来的，可以边收边推、实现「边生成边出字」。
- **result**：只有**一行** NDJSON，这一行里的 `result` 字段就是**整段正文**（例如 507 字、967 字一次性都在这一行里）。  
  也就是说，**result 不是「result 块 1、result 块 2…」**，而是**一条消息 = 一整段**。我们收到 result 时，手里已经是完整字符串，没法再拆成「流式」的小块——协议本身就没有 result 的增量格式。

所以：「最后一条完整回复」**不能**以流式方式从上游拿到，因为上游给我们的就是 **result 这一整块**。要「流式」只能靠前面的 assistant 块；若选择「只推 result」，就只能是「等完再一次性推这一块」。

---

## 等的这段时间里能做什么？（先返回 / catch）

可以。若采用「不推 assistant、只等 result 再推」的策略，在等的这段时间里可以：

1. **先给客户端一条「占位」**  
   例如立刻推一条 SSE：`content: "正在处理，请稍候…"` 或 `"后台处理中"`，前端先显示这句；等 result 到了再推**真正内容**（可替换或追加，取决于前端怎么渲染）。这样用户不会面对「完全空白在转圈」。

2. **Catch / 中间回复**  
   在桥里或上游可以做「超时/异常 catch」：例如超过 N 秒还没 result，先推一条「处理时间较长，请稍候或重试」；或把已收到的 assistant 块做一次汇总再推出去（相当于 fallback 成「不等 result，用已收到的内容先回复」）。  
   也可以做「进度类」推送：例如解析到 `tool_call` 时推一条「正在调用工具…」，前端就能显示状态而不是干等。

这些都不改「result 是整段」的事实，但能改善「只推 result」时的等待体验；实现上就是：在流式管道里多推几种类型的 SSE 块（占位、状态、最终 result）。

## 重复不是桥「拼接」出来的；type 之间是解耦的

- **桥不会把「相同内容拼两遍」**：我们只有一个 `streamedText`，且**只有在我们真正 push 出一段内容时**才往上面追加（`streamedText += textToPush`）。没有「把 assistant 和 result 两条拼成一条再发」的逻辑，也不会因为 type 不同就复制一份内容。所以**重复不是我们拼接导致的**。
- **不同 type 互相解耦**：我们按行处理，每行只认一个 type（assistant / result / message），从对应字段取正文，取完就和 `streamedText` 做比较、决定推不推；**不会把不同 type 的内容混在一起再输出**。所以**不是「不同 type 在我们这里被合并成重复」**。
- **assistant 和 result 会天然带重复吗？会，是上游行为**：Cursor Agent 常见做法是——先流式发多行 `type: assistant`（或 message），最后再发一行 `type: result` 带整段或同一段正文。所以**同一轮回复里，assistant 和 result 经常是同一段话**；重复是**上游（Cursor）在同一轮里用不同 type 或同一 type 多行发了相同/相似内容**。我们做事件级去重，正是为了在这种情况下只出一遍、不把 result 或第二段 assistant 再推出去。

## 用到的数字（当前值）

| 含义 | 当前值 | 出现位置 |
|------|--------|----------|
| **宽松判定参与的最小长度** | **8** | 当前行 / 已发送内容 ≥8 字才做「宽松归一化 + 包含判断」 |
| **宽松判定用的「已发送尾巴」长度** | **600** | 只取 `streamedText` 的**末尾 600 字**做宽松比较，不整段做正则（控性能） |
| 单行内去重：参与的最小长度 | 80 | 要推的字符串长度 < 80 不做「结尾 L 字 = 开头 L 字」检测 |
| 单行内去重：重复段长度 L 的范围 | 40～min(80, 半长) | 只把「至少 40 字且结尾=开头」的整段当重复删掉 |

## 每行 NDJSON 的处理顺序（只对有正文的 assistant/result/message）

**前提**：已从该行抽出正文 `text`，且不是 thinking/心跳。下面只写「去重相关」的步骤。

1. **事件级去重（整行跳过）**  
   用**严格归一化** `norm = text.replace(/\s+/g, ' ').trim()` 得到 `normNew`、`normStreamed`（已发送全文）。  
   若下面**任一**为真，则**整行不推、不更新 streamedText**，直接处理下一行：
   - `normNew === normStreamed`
   - `normNew.length >= 20` 且 `normStreamed.endsWith(normNew)`
   - `normStreamed.startsWith(normNew)`
   - `normNew.length >= 10` 且 `normStreamed.includes(normNew)`
   - `normStreamed.length > normNew.length` 且 `normStreamed.includes(normNew)`
   - `normNew.length >= 20` 且 `normStreamed.length >= 20` 且 `normStreamed.includes(normNew)`
   - **宽松**：当前行与已发送都 **≥8 字** 时，再算 `looseNew`、`looseStreamed`（见下），若 `looseStreamed.includes(looseNew)` 也整行跳过。  
     - `looseStreamed` 只取**已发送的末尾 600 字**（`streamedText.slice(-600)`）做宽松归一化（再合并连续重复字，如「或或」→「或」），不整段做正则。

2. **跨行 trim（只推「新部分」）**  
   若未在步骤 1 被跳过：  
   - 在**严格归一化**下，找「已发送**尾部**」与「当前行**前缀**」的**最长重叠**长度 `overlapLen`。  
   - 把当前行**去掉这段重叠前缀**，得到 `textToPush`；若为空则本行不推，直接下一行。  
   - 若不为空，用 `textToPush` 继续步骤 3，并 `streamedText += textToPush`。

3. **单行内去重**  
   对**即将推送**的 `textToPush` 再做一次检测：  
   - 若长度 < 80，不改动。  
   - 否则在**严格归一化**下，若存在长度 L（40 ≤ L ≤ min(80, 半长)）使得「结尾 L 字」=「开头 L 字」，则只保留**第一段**（去掉末尾重复段）再推送；若 trim 后为空则不推。  
   - 推送后：`streamedText += 本次实际推送的内容`。

4. **0 输出时的兜底**  
   若整条流从未推过任何 content，且曾见过非空行且（有过 JSON 解析失败 或 从未见过 assistant/result/message 正文），则在结束前推一条说明文案（解析异常 / 无可展示内容）。

---

## 宽松判定里的 8 和 600（你要的两个数）

- **8**：`normNew.length >= 8` 且 `normStreamed.length >= 8` 时，才计算 `looseNew`、`looseStreamed` 并做 `looseStreamed.includes(looseNew)`。  
  即：**只有当前行和已发送都至少 8 字时，才用「宽松版」再判一次冗余**；否则不跑宽松，避免对极短内容误杀。

- **600**：算 `looseStreamed` 时不用整段 `streamedText`，只用**末尾 600 字**  
  `streamedText.length > 600 ? streamedText.slice(-600) : streamedText`，再做宽松归一化。  
  这样重复多在半段内出现时能拦到，又不会对超长全文做正则，性能可控。

---

## 若希望「有一点重就整行拦掉」

当前逻辑是：**有重叠时只推「新部分」**（步骤 2 的 trim），不会因为「有一点重」就把整行丢掉。  
若你希望改成：**只要当前行和已发送的重叠 ≥ 8 字（归一化后），就整行不推**，在 `stream-parser.js` 的 transform 里，在 `let textToPush = trimDuplicatePrefix(streamedText, text);` 后面加两行：

```js
const overlapNormLen = normalizeForEchoCheck(text).length - normalizeForEchoCheck(textToPush || '').length;
if (overlapNormLen >= 8) continue;
```

即：重叠段 ≥ 8 字就整行拦掉，不再推 `textToPush`。flush 里同样位置可加同样判断。  
更激进可改成 `if (textToPush !== text)`（只要有重叠就整行不推）。可能误伤「整段重复块后接新内容」那种只该推新部分的场景，可按需调阈值或关掉。

---

## 小结

- **8**：宽松判定参与门槛（当前行、已发送都 ≥8 字才做宽松比较）。  
- **600**：宽松判定只取已发送的**末尾 600 字**做比较，轻量、可接受误判/漏判。  
- 事件级（步骤 1）→ 跨行 trim（步骤 2）→ 单行内（步骤 3）→ 兜底（步骤 4）；要「有一点重就整行拦掉」可在步骤 2 按重叠长度或是否重叠加一个整行 skip。

## 对你本人的影响：没重复 vs 有重复，值得花多大代价？

- **没有重复时**：流式打出来的字少、好读，观察回答时不用在一大段里找「新的部分」，体验更干净。Token 你自己说了影响不大，所以**主要收益是「看回答时更省心」**，不是省成本。
- **代价**：  
  - 已付出的：一套事件级 + 跨行 trim + 单行内 + 宽松（8/600）逻辑，要维护、要调参（8、600、40、80 等），可能有误判/漏判。  
  - 再加强的边际代价：每多一层规则就多一点复杂度和误伤风险；「有一点重就整行拦」会更容易误伤正常接续。
- **建议**：若当前程度已经让「观察回答」能接受，不必再堆更多规则；若仍经常要在一大段重复里找内容，可以优先调**现有参数**（例如 8 调大/小、600 调大）或试 DEDUP_LOGIC 里那两行「重叠≥8 整行拦」，而不是再加新算法。代价控制在「改几个数 + 一两行分支」即可。

## 开源 / 社区有没有类似方法论可参考？

- **流式「内容去重」**（同一段话在一条回复里出现两遍）在通用 LLM 流式接口里**没有统一标准**。常见做法分两层：  
  - **生成侧**：用 repetition_penalty、frequency_penalty 等抑制重复，是「少产生重复」而不是「流里再删一遍」。  
  - **传输/展示侧**：SSE 层面常见的是**按 event id / Last-Event-ID 做断线重连去重**（避免同一条 event 收两次），不是对「正文内容是否重复」做判定。
- **和咱们场景最接近的**：在「中间层」（桥）对上游 NDJSON 按行做内容去重、trim 重叠，属于**代理层对上游重复的补救**，开源里很少看到现成同名方案；可类比的是：  
  - 客户端按 `message_id` 或「时间戳+内容哈希」对**整条消息**去重（避免同一条消息渲染两次），不解决「一条消息内部重复」；  
  - 某些 chat 前端会做「相邻 chunk 内容相同则只渲染一次」，和我们的跨行 trim 思路类似，但一般是前端实现、不是桥侧。  
- **结论**：没有直接可抄的「流式内容去重」标准库或 RFC；当前桥的做法（事件级 + 跨行 trim + 单行内 + 宽松尾巴 8/600）已经是一套可维护的、轻量级的补救策略，后续主要是调参和少量分支（如「重叠≥8 整行拦」），不必再对标某个具体开源项目。
